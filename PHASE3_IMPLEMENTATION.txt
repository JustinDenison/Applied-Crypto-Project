THREAT MITIGATIONS IMPLEMENTED
===============================

T1: Unauthorized Token Issuance - PBKDF2-SHA256 Password Authentication
   - Passwords stored as PBKDF2 hashes (100,000 iterations, 32-byte salt)
   - getToken() requires password parameter
   - Wrong password rejected with "Authentication failed"
   - Default admin user: admin/admin123

T2: Token Forgery/Modification - Ed25519 Signatures
   - Group server generates Ed25519 keypair on startup
   - Every token signed before transmission
   - File server verifies signature using group server's public key
   - Forged/modified tokens rejected with "Invalid token signature"

T3: Passive Network Monitoring - AES-256-GCM Encryption
   - All messages encrypted with AES-256-GCM
   - 256-bit keys, 96-bit nonces
   - Authenticated encryption (AEAD mode) prevents tampering
   - Cleartext never sent after ECDHE handshake

T4: Replay Attacks - Nonce + Timestamp Validation
   - 128-bit random nonce in every request
   - Unix timestamp validation (±60 second tolerance)
   - Server maintains nonce_cache per session
   - Duplicate nonces rejected "Nonce replay detected"

T5: Lack of Forward Secrecy - ECDHE X25519 + HKDF
   - ECDHE handshake per connection (X25519 curve)
   - Ephemeral keys generated fresh for each session
   - HKDF-SHA256 derives session key from shared secret
   - Session keys never stored or reused
   - Compromise of one session doesn't affect others

T6: Key Compromise - Session Key Rotation Policy
   - Long-term key (Ed25519): stored in groupserver_state.json
   - Session keys: automatically rotated per connection (ECDHE ephemeral)
   - If session key compromised: only that session exposed
   - If signing key compromised: rotate immediately, invalidate tokens
   - Impact analysis documented in code comments

PROTOCOL FLOW
=============

Connection Establishment:
  1. Client generates ephemeral X25519 keypair
  2. Client sends ephemeral public key (UNENCRYPTED - safe, ephemeral)
  3. Server generates ephemeral X25519 keypair
  4. Server responds with ephemeral public key + nonce
  5. Both derive identical AES-256 session key using HKDF
  6. ✓ Forward secrecy guaranteed - keys never reused

Authenticated Request (e.g., getToken):
  1. Client: creates request with nonce + Unix timestamp
  2. Client: encrypts request with AES-256-GCM session key
  3. Server: receives encrypted blob
  4. Server: decrypts with session key
  5. Server: validates nonce (not seen before) and timestamp (within ±60s)
  6. Server: authenticates user (password hash verification)
  7. Server: creates token with expiration time
  8. Server: signs token with Ed25519 private key
  9. Server: encrypts response (token + signature) with session key
  10. Client: decrypts response, verifies token received

Authorized File Operation (e.g., upload):
  1. Client: includes token + token_signature in encrypted request
  2. File Server: decrypts request
  3. File Server: validates nonce + timestamp
  4. File Server: verifies token signature (fails if tampered)
  5. File Server: checks token not expired
  6. File Server: validates user in token's groups
  7. File Server: processes file upload
  8. File Server: encrypts response, sends back
